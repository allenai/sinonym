# Sinonym Data Directory

This directory contains the data files and trained models used by the Sinonym library.

## Data Files

### Name Frequency Data
- **`familyname_orcid.csv`**: Chinese surnames with frequency data (parts per million) derived from ORCID records. Contains the most common Chinese surnames like 王, 李, 张, etc., with their frequency statistics.
- **`givenname_orcid.csv`**: Chinese given name characters with usage statistics from ORCID records. Includes character, pinyin romanization, and frequency (ppm).

### Corpus-Derived Given-Name Statistics
- **`given_position.csv`**: Per-syllable counts of romanized syllables in the first (`initial_count`) and second (`final_count`) position of two-syllable Chinese given names. Consumed by the positional feature for 3-token parses (`NameParsingService`, deadband 1.5 / cap 1.5).

It is generated by `scripts/generate_name_statistics.py` from two sources pooled together (per-row provenance in the `source` column):
1. **wainshine/Chinese-Names-Corpus 120W** (Apache-2.0, ~1.14M Han full names; download command in the generator docstring — the 12MB raw corpus is not committed). Names are split deterministically against the `familyname_orcid.csv` surname inventory (98.9% split rate) and romanized per character with pypinyin; 963,777 two-syllable given pairs.
2. **In-house byline supplement**: 247,690 distinct two-token romanized author names with exactly one hyphenated token, mined from an author-mention parquet via the hyphen convention (the hyphenated token is the given name). Distinct-name weighted and rescaled to the 120W pair total before pooling, so it both covers non-pinyin spellings ("kung", "sang" as a given initial) and keeps byline conventions alive for syllables rare in the Han corpus.

Thresholds: byline-only positional rows need >= 10 distinct names. This keeps the asset small (given_position.csv ~10KB).

Training-data provenance: this asset is trained on the 120W corpus plus byline mining only. `acl_2025_authors.txt` is **not** a training source for it (it remains eval provenance via `tests/test_acl.py`); neither source uses sinonym's own parser output as labels (only surname-inventory membership and the hyphen orthographic convention), so there is no train/eval circularity.

Attribution: the 120W corpus is from [wainshine/Chinese-Names-Corpus](https://github.com/wainshine/Chinese-Names-Corpus), Apache License 2.0.

Note: the surname-position **usage** table (`surname_usage_acl.csv`, below) intentionally remains ACL-mined. Replacing it with corpus-scale marginal counts was measured and rejected: the 1+1 mirror expectations (e.g. flip "Zhou Xun"/"Gao Wei"/"Gui Rui" but keep "Sun Ao"/"Yu Chao"/"Yu Fei") are not separable by any monotone function of marginal surname-vs-given log-odds — keep-cases sit at larger deltas with smaller margins than flip-cases (Sun Ao delta +5.90 vs Lu Xun +4.88; Yu Chao +2.92/margin 0.04 vs Gao Wei +2.96/margin 0.39). Separating them requires joint (surname, given) full-name counts at byline scale (planned stage 2).

### Surname-Position Usage Table
- **`surname_usage_acl.csv`**: Per-syllable surname-position vs given-position counts mined from `acl_2025_authors.txt` by `scripts/generate_surname_usage_data.py`. Feeds the 1+1 usage log-odds feature (hinge 2.25).

### Romanization-Conditional Surname Table
- **`surname_romanizations.csv`**: Per-spelling surname shares keyed by as-written (norm_light) romanized spelling. Generated by `scripts/generate_surname_romanizations.py` from the repo's own romanization rule tables; consumed by the surname-scoring discount in `NameParsingService`.
  - `spelling`: as-written romanized spelling (lowercase, diacritics/hyphens stripped — a `norm_light` fixed point).
  - `mandarin_target`: the Mandarin pinyin surname whose frequency mass the spelling reaches through romanization remapping (or through the `CANTONESE_SURNAMES` alias).
  - `surname_ppm_as_written`: estimated surname frequency (ppm) the spelling carries *as written*: the full target mass for attested romanizations, `target mass x target_share` for penalty spellings.
  - `target_share`: share of the remap target's surname mass the spelling keeps when scored as a surname. `1.0` for attested surname romanizations (`CANTONESE_SURNAMES` keys, curated Wade-Giles/Taiwanese surname spellings such as `tsai`, `kuo`, `hsieh`); `~e^-4` (the measured operating point) for spellings that inherit surname mass only via remapping with zero as-written attestation (e.g. `fai` -> `hui`, `kung` -> `gong`).
  - **Korean-dominant full-share exclusions** (`KOREAN_DOMINANT_FULL_SHARE_EXCLUSIONS` in the generator): five `CANTONESE_SURNAMES` keys whose as-written spelling is overwhelmingly Korean — `jung` (정), `moon` (문), `im` (임), `kyeong` (경), `pak` (박) — are demoted from full share to the penalty share. Granting them their Mandarin target's full mass (zheng/wen/lin/jing/bai) over-credited Korean surname-first names in routing (net -pp_vys on cases like "Jung Im Na"/"Jung Mi Cha"/"Pak Siau Wei"). They stay attested as-written but score at the discounted mass (a rare Chinese surname) rather than the full target. Their real-but-minor Chinese usage keeps the penalty share instead of falling to nothing. Heavily Chinese-used overlap spellings stay full share (`lee` 李, `lim` 林, `chang`/`jang` 张, `choi` 蔡, `ho` 何, `han` 韩, `yu` 于/俞, `koo` 顾/古, plus bicultural `shin`/`son`/`soo`/`suh`); the per-spelling measured rationale lives in the generator. `jang` in particular would otherwise be trimmed but is kept full share because the `{im, jang, jung}` trio flips the borderline eval case "Li Zheng" via a global surname-rank shift, and `koo` because trimming it rejects "Koo Ming"; the parser eval must hold at its 17-name baseline.
  - The scoring hook reads `spelling` and `target_share` and (via initialization) loads each row's `surname_ppm_as_written` into the runtime surname frequency table, so full-share rows carry the target's mass and penalty CANTONESE keys carry the discounted mass. Corpus-derived estimates (e.g. ORCID with affiliation country) can replace the numbers without a schema change.
### Chinese vs Japanese Name Classifier 
- **`chinese_japanese_classifier.joblib`**: Pre-trained scikit-learn model for distinguishing Chinese from Japanese names when written in Chinese characters. 
- **`model_features.json`**: Feature vocabulary for the Chinese vs Japanese classifier, containing character n-grams and linguistic patterns used by the model.

**Model Details:**
- **Purpose**: Classify all-Chinese character names (like "山田太郎") as Chinese vs Japanese
- **Accuracy**: 99.53% on test set of 263,409 names  
- **Training Data**: 1.3M names (1.14M Chinese, 172K Japanese from open corpora)
- **Features**: 5,000 character n-grams + 20 linguistic heuristic features
- **Algorithm**: Logistic Regression with TF-IDF character features
- **Use Case**: Fills gap where Japanese names in Chinese characters bypass romanization-based detection
- **Status**: **ACTIVE - Integrated in `sinonym.services.ml_japanese_classifier.MLJapaneseClassifier`**

**Model Features:**
- Character-level 1-3 gram TF-IDF features (5,000 features)
- Linguistic binary flags (20 features):
  - Japanese iteration mark (々)
  - Japanese/Chinese surname character patterns
  - Japanese/Chinese name ending patterns
  - Unique character indicators
  - Name length patterns
  - Character frequency patterns
s
**Training Date**: 2025-07-30

## Model Integration

The Chinese vs Japanese classifier is integrated into `sinonym.services.ethnicity.EthnicityClassificationService` and automatically activates for all-Chinese character inputs that pass initial ethnicity screening.

If ML dependencies are not available, the system gracefully falls back to the existing rule-based ethnicity classification without the enhanced Japanese detection capability.

## Data Generation Scripts

Scripts in the `scripts/` directory are used to generate and process these data files:
- `train_ml_classifier_for_chinese_vs_japanese.py`: Trains the Chinese vs Japanese classifier model
- `generate_surname_romanizations.py`: Regenerates `surname_romanizations.csv` from the romanization rule tables
- `generate_name_statistics.py`: Regenerates `given_position.csv` from the 120W corpus plus the byline supplement (source paths are CLI arguments)
- `generate_surname_usage_data.py`: Regenerates `surname_usage_acl.csv` from `acl_2025_authors.txt`
